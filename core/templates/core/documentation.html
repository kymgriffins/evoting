Overview

The EVoting System is a Django-powered platform for student council elections with integrated manifesto tracking. Built with Python, Django 6.0, SQLite, and Django REST Framework.

Three user roles: Admin (manages elections), Candidate (uploads manifestos + progress updates), Voter (casts votes + rates manifestos).

Key features:

  • Secure electronic voting with double-vote prevention
  • Manifesto tracking — candidates post progress updates on promises
  • Public leaderboard ranked by promise success rate
  • Audit logging — every action is recorded (transparency without blockchain)
  • REST API — all data available as JSON
  • Responsive design — works on mobile and desktop
School project note: This is designed for the Catholic University of Eastern Africa (CUEA) Student Council elections but can be adapted for any organization.

Quick Start

1
Clone & install 
git clone https://github.com/kymgriffins/evoting.git
cd evoting
python -m venv venv
source venv/bin/activate  # Linux/Mac
pip install -r requirements.txt
2
Run migrations & seed data 
python manage.py migrate
python manage.py populate_cuea
3
Start the server 
python manage.py runserver

Visit http://localhost:8000 and use the Quick Test Login buttons on the login page.

Models (Database Schema)

The app has 9 database tables. Here's how they connect:

User

Extends Django's built-in user with:

id username password email role student_id is_verified

Roles: voter, candidate, admin

Election

id title description start_date end_date is_active created_at

A voting event with a time window.

Position

id title description order election (FK)

E.g. President, Vice President, Secretary, Treasurer.

Candidate

id user (OneToOne) position (FK) election (FK) bio is_approved

A person running for office. Must be approved by admin.

Manifesto

id candidate (FK) title description category

A campaign promise. Categories: Education, Healthcare, Infrastructure, Student Welfare, Other.
Immutable — once created, can only be updated via progress updates.

Vote

id voter (FK→User) candidate (FK) position (FK) election (FK) timestamp

unique_together = [voter, position, election] prevents double voting.

ManifestoUpdate

id manifesto (FK) status description evidence_url created_at

Progress update: Not Started, In Progress, Completed, Delayed.

ManifestoRating

id user (FK) manifesto (FK) rating (1-5) comment

One rating per user per manifesto.

AuditLog

id user (FK, nullable) action details ip_address timestamp

Records every important action for transparency. Replaces blockchain.

Views (How Pages are Served)

The app uses function-based views in a single core/views.py file. Each URL maps to a view function that:

  1. Receives the HTTP request

Templates (HTML UI)

All templates use Django's template inheritance. Every page extends base.html which provides:

  • Navigation bar with role-based menu items
  • Flash message display (success/error popups)
  • Footer
  • Responsive CSS with CSS variables

Migrations (Database Changes)

Migrations are Django's way of propagating changes to the database schema. They're like version control for your database.

Existing migrations

core/migrations/
├── 0001_initial.py         # All tables except ManifestoRating
├── 0002_manifestorating.py # Added ManifestoRating table
└── __init__.py

How to run migrations

1
Make changes to models.py

Add/edit fields in core/models.py.

2
Create migration file 
python manage.py makemigrations

This auto-generates a migration file in core/migrations/.

3
Apply migration 
python manage.py migrate

This updates the actual database.

4
Check migration status 
python manage.py showmigrations

Shows which migrations have been applied ([X]) and which are pending ([ ]).

Reset database: Delete db.sqlite3 and re-run python manage.py migrate && python manage.py populate_cuea to start fresh.

Rollback a migration

python manage.py migrate core 0001
python manage.py migrate core 0002

Replace 0001 with the migration number you want to go back to.

Check what SQL a migration will run

python manage.py sqlmigrate core 0002

Preview the SQL without applying it.

Mock Test Presentation

Use this test flow to demonstrate the app in a presentation. Each step includes credentials and expected outcomes.

Test Flow 1: Admin — Create an Election

  1. Go to /login/ and click Admin (or use admin / admin123)
  2. Click Elections in the nav
  3. Fill in title, description, set start/end dates (future dates)
  4. Select "Clone positions from" = "CUEA Student Council Elections 2026"
  5. Click Create Election
  6. Expected: Election created with 4 positions (President, VP, Secretary, Treasurer)

Test Flow 2: Candidate — Post Manifesto Updates

  1. Go to /login/ and click Candidate (or cand001 / cand123)
  2. Click Manifestos in the nav
  3. Click Updates on any manifesto
  4. Set status = "In Progress", write a description, click Post Update
  5. Expected: Update appears, audit log records it

Test Flow 3: Voter — Cast Vote

  1. Go to /login/ and click Voter (or v0001 / voter123)
  2. Click Dashboard, then Cast Your Vote on the active election
  3. Select a candidate for each position (cards highlight green with checkmark)
  4. Click Review & Cast Vote
  5. Review selections in the modal, click Confirm & Cast Votes
  6. Expected: Success message, dashboard shows "Voted" badge

Test Flow 4: Voter — Rate Manifestos

  1. Login as voter (same as above)
  2. Go to Tracking in the nav
  3. Click Rate on any manifesto
  4. Select stars and optional comment, submit
  5. Expected: Rating shows in the Leaderboard

Test Flow 5: Admin — View Audit Trail

  1. Login as Admin
  2. Click Audit in the nav
  3. Use the search bar to filter by action type
  4. Expected: Every vote, registration, and update is timestamped with IP

Test Flow 6: Leaderboard & Results (Public)

  1. Visit /leaderboard/ — see candidates ranked by success rate
  2. Visit /results/ — see past election results with vote counts
  3. Visit /tracking/ — see all manifestos with latest status
  4. Expected: No login required for these pages
